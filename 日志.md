### 详细分析

#### Bug #18: JavaScript `>>>` 位移取模陷阱（核心）

```js
// 错误写法：JS 中移位量会对 32 取模
for (var i = 0; i < 8; i++) msg.push((ml >>> (56 - i * 8)) & 0xff);

// i=3 时：56-24=32, 24 >>> 32 = 24 >>> 0 = 24 ← 错！应该 0
// i=0 时：56-0=56, 56%32=24, 24 >>> 56 = 24 >>> 24 = 0 ← 碰巧对
```

**JS 位运算黄金法则：** 移位量 >= 32 时，JS 会对 32 取模后再移。SHA1 的 64 位消息长度编码中 `>>> 32`、`>>> 40`、`>>> 48`、`>>> 56` 全部受影响。

**正确写法：** 显式处理 64 位值的高/低 32 位：
```js
msg.push(0, 0, 0, 0);                           // 高 32 位（消息 < 4GB 恒为 0）
msg.push((ml >>> 24) & 0xff, (ml >>> 16) & 0xff, (ml >>> 8) & 0xff, ml & 0xff);  // 低 32 位
```

#### Bug #19: JS 32 位有符号整数

JS 所有位运算（`&` `|` `^` `~` `<<` `>>` `>>>`）先将操作数转为 **32 位有符号整数**。`~b` 在有符号模式下 `~0xEFCDAB89 = -0xEFCDAB8A`，可能产生非预期结果。解决方案：关键路径加 `>>> 0` 强制无符号。

## 二、架构决策

### 为什么不用后端生成 Token？

微信小程序无 `crypto` 模块，业界常用云函数生成 HMAC 签名。但本方案选择纯 JS 实现 SHA1+HMAC，原因：
- 避免依赖云服务，离线可用
- SHA1 算法 ~100 行 JS，体积小
- Token 24h 缓存，实际每分钟只用算一次

### Token 缓存策略

```js
// 首次请求生成 token，24h 后过期
// 过期前 5 分钟自动刷新
if (!cachedToken || now > tokenExpiry - 300) {
  cachedToken = crypto.generateToken(...);
  tokenExpiry = now + 86400;
}
```

### 乐观更新 vs 等待确认

用户点击开关 → 立即更新 UI（乐观）→ 发送 API → 1.5s 后刷新确认。这样用户看到即时响应，避免 2-3 秒的空白等待。

|------|------|------|
| Base URL | `api.heclouds.com` | `iot-api.heclouds.com` |

## 四、下行控制完整链路

```
[微信小程序]
  ↓ POST /thingmodel/set-device-property {params:{win:false}}
[OneNET 平台]
  ↓ MQTT 下行: $sys/{pid}/{device}/thing/property/set
  ↓ {"id":"xxx","version":"1.0","params":{"win":{"value":false}}}
[EG800 4G 模块]
  ↓ UART AT 指令
[ESP32-S3]
  ↓ g_win_state = false
  ↓ OLED 更新 / LED 更新
  ↓ 回复: $sys/{pid}/{device}/thing/property/set_reply {"id":"xxx","code":200,"msg":"success"}
```

**当前状态：** 链路前两步（小程序→OneNET）已验证通过。第三步起（设备端）需要 ESP32 在线并正确处理 `thing/property/set` Topic 消息。API 返回 `code:10411`（设备响应超时）说明平台已下发指令但设备未回复 ACK。
