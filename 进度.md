## 一、本阶段突破

### 1.1 按键状态同步上报

物理按键（GPIO6）按下松手时同步切换 `win` 属性状态，随 5s 周期上报 OneNET：

| 状态 | OLED | win |
|------|------|:---:|
| 初始 | 温度 | `true` 开 |
| 按一下 | 我早已麻痹 | `false` 关 |
| 再按一下 | 温度 | `true` 开 |

`g_win_state` 初值设为 `true`（与 OLED 初始 ON 对齐），按键松手时通过 `Key_GetNum()` 触发 toggle。

### 1.2 断连自动重连

主循环检测 `!EG800_IsConnected()` → 自动断电重启模块 → 完整重新初始化。

触发源（三处自动检测）：
- `EG800_Publish` 读到 `ERROR` → 标记断连
- `EG800_Publish` 读到 `RDY`（模块异常重启）→ 标记断连
- URC 任务读到 `RDY` → 标记断连

### 1.3 微信小程序 v2.0

基于 OneNET 新平台 API（`iot-api.heclouds.com`）重构，实现数据展示+下行控制双功能：

**核心功能：**
- 每 5 秒轮询 `query-device-property` 获取温度和 win 状态
- 点击卡片按钮通过 `set-device-property` 下发 win 开关指令
- 乐观更新 UI（指令发送成功后立即更新，1.5s 后确认）
- HMAC-SHA1 Token 自动生成，24h 有效期缓存

**技术栈：**
- 鉴权：HMAC-SHA1 签名 token（纯 JS 实现，无需后端）
- 查询：`GET /thingmodel/query-device-property?product_id=...&device_name=...`
- 下行：`POST /thingmodel/set-device-property` {product_id, device_name, params}
- 响应格式：`[{identifier, time, value, data_type, access_mode}]`

**文件结构：** 见第四节

## 三、核心 Bug 修复记录

共修复 **16 个错误**，详情见「错误示范.md」：

| # | 问题 | 症状 | 修复 |
|:---:|------|------|------|
| 1 | Topic 协议混用 | `+QMTSTAT: 0,1` | 统一物模型协议 |
| 2 | JSON/Topic 不配套 | 服务器断连 | dp 格式配 $dp, params 格式配 thing |
| 3 | read_line 等 `>` | 超时 | 逐字节 `uart_read_bytes` |
| 4 | URC 任务抢 UART | `RX[0]:` | `g_publish_busy` volatile flag |
| 5 | 栈溢出 | 不停重启 | 大数组改为 `static` |
| 6 | `\"` 转义失败 | `RX[0]:` | 用 `AT+QMTPUBEX` 两步式 |
| 7 | `#` 通配符订阅 | ERROR | 使用精确 topic |
| 8 | 登录后不订阅 | 几秒断连 | 登录后立即 `QMTSUB` |
| 9 | tick 清零位置 | JSON id=0 | 移到 `snprintf` 之后 |
| 10 | 测试数据无效 | 断连 | 发有效 JSON |
| 11 | URC 超时 5000ms 太长 | publish 丢 `>` | 改为 200ms 逐次检查 flag |
| 12 | QMTCFG 重置回显 | 模块回显混入 RX | 订阅后重发 `ATE0` |
| 13 | 模块异常重启送 RDY | publish 等不到 `>` | 检测 RDY → 标记断连 |
| 14 | 断连后 g_connected 未更新 | 反复 ERROR 循环 | publish 检测 ERROR → 标记断连 |
| 15 | SIM 初始化未完成 | `+CME ERROR: 14` | CPIN? 重试 3 次，间隔 3s |
| 16 | 按键未同步 win | win 永远显示关 | 按键 toggle `g_win_state` |
