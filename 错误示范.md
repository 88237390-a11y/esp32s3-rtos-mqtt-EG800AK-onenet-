# ESP32 + EG800 OneNET MQTT 连接 — 错误示范

**验证日期：** 2026-07-14  
**目的：** 记录所有踩过的坑，避免重复犯错

---

## 错误 1：Publish 和 Subscribe Topic 协议混用

### 错误表现
```
Publish:  $sys/yS8CG4x6L5/text/dp/post/json
Subscribe: $sys/yS8CG4x6L5/text/thing/property/set
```
→ OneNET 返回 `+QMTSTAT: 0,1`（服务器断开连接）

### 原因
OneNET MQTT 有两种协议：物模型协议 (`thing/property/...`) 和 dp 数据点协议 (`dp/post/json`)。同一个连接里**不能混用**两种协议的 topic。上行和下行必须属于同一套协议。

### 正确做法
统一用物模型协议：
```
Publish:  $sys/yS8CG4x6L5/text/thing/property/post
Subscribe: $sys/yS8CG4x6L5/text/thing/property/set
```

## 错误 2：JSON 格式与 Topic 不匹配

### 错误表现
Topic 用了 `$dp`（旧版数据点协议），但 JSON 用了物模型格式：
```json
{"id":"25","version":"1.0","params":{"heat":{"value":37}}}
```
→ OneNET 解析失败，断开连接

### 原因
- `$dp` topic = 旧版协议，数据格式是 `{"id":123,"dp":{"heat":[{"v":37}]}}`
- `thing/property/post` topic = 物模型协议，数据格式是 `{"id":"25","version":"1.0","params":{"heat":{"value":37}}}`

Topic 和 JSON 格式**必须配套**，不能张冠李戴。

## 错误 3：用 `eg800_read_line` 等 `>` 提示符

### 错误代码
```c
eg800_send_cmd(cmd);
// 错误！eg800_read_line 等 \n 才返回，但 > 后面没有 \n
if (eg800_read_line(line, sizeof(line), 3000) != ESP_OK || line[0] != '>') {
    ESP_LOGE(TAG, "Publish: no '>' prompt");
}
```

### 为什么错
`AT+QMTPUBEX` 返回的是 `> `（大于号+空格），**不带 `\r\n`**。`eg800_read_line` 必须等到 `\n` 才返回 → 永远超时。

### 正确代码
```c
// 逐字节读，看到 > 就停
char ch = 0;
TickType_t deadline = xTaskGetTickCount() + pdMS_TO_TICKS(5000);
while (xTaskGetTickCount() < deadline) {
    if (uart_read_bytes(EG800_UART, (uint8_t *)&ch, 1, pdMS_TO_TICKS(200)) > 0
        && ch == '>') {
        break;
    }
}
if (ch != '>') { /* 失败 */ }
```

## 错误 4：URC 任务和 Publish 函数抢 UART

### 错误表现
URC 监听任务 (`eg800_urc_task`) 和 `EG800_Publish` 都在读 UART，导致 `AT+QMTPUBEX` 发出去后，`>` 被 URC 任务抢先读走。

日志表现：`RX[0]:` — publish 函数一个字都没读到

### 为什么错
ESP32-S3 是**双核** CPU。`vTaskSuspend()` 不是瞬时的——如果 URC 任务刚好在另一个核上执行 `uart_read_bytes`，数据到达时它可能先被唤醒，抢走 `>`。

### 正确做法
用 `volatile bool g_publish_busy` flag：
```c
// URC 任务中
while (g_connected) {
    if (g_publish_busy) {
        vTaskDelay(pdMS_TO_TICKS(50));
        continue;  // 不读 UART
    }
    eg800_read_line(...);
}

// EG800_Publish 中
g_publish_busy = true;   // 阻止 URC 任务
// ... publish ...
g_publish_busy = false;  // 允许 URC 任务恢复
```

## 错误 5：大数组放栈上导致栈溢出

### 错误代码
```c
esp_err_t EG800_Publish(const char *json_str)
{
    char escaped[512];  // 512 bytes
    char cmd[768];      // 768 bytes
    // = 1280 bytes 全压在 FreeRTOS 任务栈上
    // → vApplicationStackOverflowHook → 重启
}
```

### 错误日志
```
Backtrace: ... vApplicationStackOverflowHook ...
--- 0x4037c916: vApplicationStackOverflowHook ...
ELF file SHA256: ...
Rebooting...
```

### 正确做法
声明为 `static`，移到 BSS 段：
```c
static char cmd[512];
```

## 错误 6：AT+QMTPUB 一步式 \" 转义

### 错误代码
```c
// 想用 AT+QMTPUB 一步发送，JSON 双引号用 \" 转义
snprintf(cmd, sizeof(cmd),
         "AT+QMTPUB=%d,0,0,0,\"%s\",\"%s\"",
         idx, topic, escaped_json);
```

### 为什么错
移远模块的 AT 命令解析器**不支持 `\"` 转义**。当解析到 JSON 中的 `\"` 时，模块误认为字符串已结束，后续内容被当作非法参数丢弃。

日志表现：`RX[0]:` — 模块完全不响应

### 正确做法
用 `AT+QMTPUBEX` 两步式——JSON 数据是**原始二进制发送**的，不需要任何转义：
```
AT+QMTPUBEX=0,0,0,0,"topic",<length>  → 模块回 >
→ 直接发原始 JSON（不转义）
```

## 错误 7：订阅用了 `#` 通配符

### 错误代码
```c
snprintf(g_topic_sub, sizeof(g_topic_sub),
         "$sys/%s/%s/#", g_product_id, g_device_name);
// → AT+QMTSUB=0,1,"$sys/yS8CG4x6L5/text/#",0 → ERROR
```

### 为什么错
`AT+QMTSUB` 使用 `#` 通配符时，移远 EG800 模块可能返回 ERROR。模块在 OneNET 模式下对通配符订阅有限制。

### 正确做法
使用**精确 topic**：
```c
snprintf(g_topic_sub, sizeof(g_topic_sub),
         "$sys/%s/%s/thing/property/set", g_product_id, g_device_name);
```

## 错误 8：login 后不订阅直接 publish

### 错误表现
MQTT 登录成功，但不订阅。7-8 秒后 `AT+QMTPUBEX` 返回 `+QMTSTAT: 0,4` 或 `+QMTSTAT: 0,1`。

### 原因
OneNET 要求设备登录后发送订阅，没有订阅的连接会被服务器在几秒内关闭。

### 正确做法
MQTT 登录成功后**立即订阅**下行 topic，再进入主循环发布。

## 错误 9：`publish_tick = 0` 写错位置

### 错误代码
```c
if (EG800_IsConnected() && ++publish_tick >= 25) {
    publish_tick = 0;  // ← 在 snprintf 之前！
    snprintf(json, sizeof(json), "{\"id\":%d,...}", publish_tick, ...);
    //                                     ↑ id 永远是 0
}
```

### 正确代码
```c
if (EG800_IsConnected() && ++publish_tick >= 25) {
    snprintf(json, sizeof(json), "{\"id\":%d,...}", publish_tick, ...);
    //                  ↑ 先拿到值（25、50、75...）
    EG800_Publish(json);
    publish_tick = 0;  // ← 发完再清零
}
```

## 错误 10：不验证 `AT+QMTPUBEX=?` 就直接用

### 建议
在 `eg800_check_alive` 阶段加两行查询，确认模块固件支持这两个命令：

```c
eg800_send_cmd_expect("AT+QMTPUB=?", "OK", 2000);
eg800_send_cmd_expect("AT+QMTPUBEX=?", "OK", 2000);
```

如果返回 ERROR，说明模块固件不支持 MQTT 发布功能，需要升级固件。

## 错误 11：URC 任务 read_line 超时 5000ms 太长

### 错误表现
Publish 时偶尔出现 `no '>' prompt`，`>` 被 URC 任务抢走。日志中模块回显缺 `A`（如 `T+QMTPUBEX=...`）。

### 原因
URC 任务中 `eg800_read_line(timeout=5000)` 一次阻塞 5 秒。即使 publish 任务设了 `g_publish_busy = true`，URC 任务卡在 `read_line` 内部看不到 flag，仍然在读 UART，抢走 `>` 和模块回显字节。

### 正确做法
```c
// URC 任务中：超时从 5000ms 改为 200ms
if (eg800_read_line(line, sizeof(line), 200) == ESP_OK) {
```
200ms 足够接收完整一行 URC（115200 波特率下 200 字节仅需 ~17ms），同时最多 200ms 就检查一次 `g_publish_busy` flag，publish 开始后迅速让出 UART。

## 错误 12：QMTCFG 系列命令重置 ATE0 回显

### 错误表现
连接成功后 publish 的 RX 日志里出现完整的 AT 命令回显：
```
RX[75]: AT+QMTPUBEX=0,0,0,0,"$sys/.../thing/property/post",80
ERROR
```

### 原因
`eg800_check_alive` 阶段发了 `ATE0` 关回显，但后续 `eg800_mqtt_config` 中连续多条 `AT+QMTCFG` 命令可能导致模块内部状态重置，回显重新开启。

### 正确做法
MQTT 订阅成功后、设 `g_connected = true` 之前，补一发 `ATE0`：
```c
eg800_send_cmd_expect("ATE0", "OK", 1000);
g_connected = true;
```

## 错误 13：模块异常重启发送 RDY 导致 publish 失败

### 错误表现
```
I (28017) EG800: TX: AT+QMTPUBEX=...
I (33074) EG800: RX[7]: RDY
E (33074) EG800: Publish: no '>' prompt
```

### 原因
EG800 模块在 LTE 发射时瞬时电流可达 2A+，若供电不足会掉电复位。复位后模块发送 `RDY`（上电就绪信号），此时 MQTT 连接已全部丢失。`g_connected` 仍为 true，publish 继续尝试发送，但模块处于初始状态，不会回 `>`。

### 正确做法
三处加 RDY/ERROR 检测：
1. **EG800_Publish** — 逐字节读响应时，若读到 `RDY` 或 `ERROR`，设置 `g_connected = false`
2. **URC 任务** — 若读到 `RDY`，设置 `g_connected = false` 并退出
3. **主循环** — 检测到 `!EG800_IsConnected()` 后自动触发重连

```c
// publish 中
if (strstr(buf, "ERROR")) {
    g_connected = false;
} else if (strstr(buf, "RDY")) {
    g_connected = false;
}
```

硬件建议：EG800 VCC 与 GND 间并联 ≥1000μF 电解电容，扛住 LTE 发射尖峰电流。

## 错误 14：断连后 g_connected 未更新导致反复 ERROR

### 错误表现
```
E (134155) EG800: Publish: no '>' prompt
I (139667) EG800: TX: AT+QMTPUBEX=...   ← 还在重试
I (144734) EG800: RX[7]: ERROR
E (144734) EG800: Publish: no '>' prompt
I (149734) EG800: TX: AT+QMTPUBEX=...   ← 无限循环
```

### 原因
MQTT 连接断开后，模块对 `AT+QMTPUBEX` 返回 `ERROR`，但代码只判断 `>`，没处理 `ERROR`。`g_connected` 保持 true，主循环每 5 秒就发一次 publish，每次都失败。

### 正确做法
见错误 13 的正确做法 — publish 中检测 `ERROR` 后立即 `g_connected = false`，阻止后续无效重试。

## 错误 15：SIM 初始化未完成 — CME ERROR 14

### 错误表现
```
I (26927) EG800: Checking SIM...
E (29966) EG800: Timeout waiting 'READY' for cmd: AT+CPIN? (last: +CME ERROR: 14)
E (29966) EG800: SIM not ready
```

### 原因
模块冷启动后 SIM 卡需要额外时间初始化。`+CME ERROR: 14` 表示 "SIM busy"，是瞬态错误，通常几秒后自动恢复。原代码只检测一次就放弃。

### 正确做法
SIM 检测加重试（3 次，间隔 3 秒）：
```c
for (int i = 0; i < 3; i++) {
    eg800_flush_rx();
    if (eg800_send_cmd_expect("AT+CPIN?", "READY", 5000) == ESP_OK) break;
    vTaskDelay(pdMS_TO_TICKS(3000));
}
```

## 错误 16：按键状态未同步到 win 属性

### 错误表现
OneNET 平台 win 属性始终显示"关"，物理按键按下后 win 数据不变。

### 原因
代码中 `Key_GetNum()` 只切换了 OLED 显示 (`oled_on`)，但没有更新 `g_win_state`。`g_win_state` 只在下行指令回调 `on_downlink` 中被修改，物理按键完全不参与。

### 正确做法
按键松手时同步 toggle `g_win_state`：
```c
if (Key_GetNum()) {
    g_win_state = !g_win_state;  // 同步切换 win 状态
    oled_on = !oled_on;
}
```
且 `g_win_state` 初始值设为 `true`（与 OLED 初始 ON 对齐），确保"温度显示=开，我早已麻痹=关"。
